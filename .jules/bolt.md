## 2025-02-17 - Avoid O(N) array scans inside rendering map loops
**Learning:** Writing inline `.filter(el => el.key === groupKey).length` or `.reduce` operations inside a `.map` loop over React elements causes an implicit O(N²) scaling overhead. In `PlanComparison` tables where lists of rows scale with the number of players, these O(N) calculations running on every row caused unnecessary render blocking. Similarly, in large data processing logic like `calculateVorp`, scanning large arrays using `.filter` iteratively causes severe lag O(P*N).
**Action:** When calculating sub-group statistics or counts for nested elements like `rowSpan`, always pre-aggregate and calculate sizes in a single `useMemo` map iteration ahead of the render. This achieves O(N) grouping and O(1) lookups inside the render. For large background arrays, initialize a `Map` structure with `.get/.set` for O(1) reads rather than calling `.filter()` iteratively.
